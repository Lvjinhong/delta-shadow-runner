# Delta External Vision Runner 重构计划

## 1. 意图

交付一个纯 Python 的 Windows 外部视觉自动化脚本。系统只通过桌面截图观察目标窗口，使用模板、局部特征、OCR 和几何规则估计状态，通过路线规划与行为状态机生成动作，并仅使用 Windows 标准用户态输入接口执行键鼠动作。

项目不包含浏览器 Dashboard、REST/WebSocket 控制面或模拟数据平台。早期 TypeScript A* 的数值与图结构回归已迁移到 Python，旧平台只保留在 Git 历史中。

## 2. 完成标准

以下项目必须全部有当前 Windows 开发机上的运行证据，才允许标记完成：

1. **源码调研**：对候选 GitHub 项目的 README、License 和关键实现代码进行交叉核验，形成带链接、代码锚点、维护状态和取舍理由的调研报告。
2. **屏幕采集**：在 Windows 上从目标显示器或指定窗口区域持续抓帧；1080p 基线达到至少 20 FPS，记录平均值、P95 帧延迟和丢帧数。
3. **数据与回放**：支持录制帧、时间戳、分辨率和人工动作；支持不依赖在线目标程序的确定性离线回放，回放结果可复现。
4. **视觉感知**：至少识别运行阶段、HUD/交互提示、方向或视觉锚点，并输出置信度和可解释诊断图。准确率必须由独立标注样本集计算，不能凭主观截图宣称。
5. **定位与路线**：把感知结果转换为当前路线节点或局部位姿，使用固定 waypoint/A* 选择到目标的路线；偏离和低置信度时停止或重定位。
6. **动作规划**：状态机能够生成前进、转向、交互、停止和恢复动作；卡住检测必须来自连续画面变化或定位进度，而不是模拟事件。
7. **标准输入**：提供 dry-run 和 armed 两种模式；armed 只允许 Windows 标准用户态输入，必须校验前台窗口、最大按键时长、急停键和人工接管。
8. **一键运行**：Windows 一键入口完成依赖检查、配置检查、采集基线、自检和日志目录创建；失败必须给出可定位的错误，重复执行不得启动重复 Worker。
9. **受控 E2E**：先在确定性测试窗口跑通截图→识别→规划→输入→到达目标；再在用户授权的受控游戏场景中跑通至少一条固定路线。两类 E2E 都必须由独立评估者复跑。
10. **最终审计**：本地、GitHub、Windows 提交一致且工作区干净；测试、构建、安全检查和运行产物均有可追溯记录。

## 3. 阶段验收指标

### 3.1 采集

- 1920×1080 主基线：平均帧率 `>= 20 FPS`。
- P95 抓帧延迟：`<= 50 ms`。
- 连续 60 秒采集：崩溃、尺寸漂移和未解释丢帧均为 `0`。

### 3.2 感知

- 场景状态分类：独立验证集准确率 `>= 95%`。
- 关键视觉锚点：precision、recall、F1 分别报告，F1 目标 `>= 0.90`。
- 所有低于阈值的输出必须转入 `uncertain`，不得继续发送移动输入。

### 3.3 路线与控制

- 离线回放的路线节点序列可重复，重复运行结果一致。
- 受控测试窗口固定路线成功率：至少 `20/20`。
- 卡住恢复测试：至少 `10/10` 能检测、停止错误动作并恢复或安全退出。
- 急停触发后 `200 ms` 内释放所有按键。

### 3.4 实际游戏场景

- 使用一个固定分辨率、固定地图区域和固定路线作为首个 MVP。
- 至少完成一条从已识别起点到目标点的受控路线；全过程保存时间戳、感知、决策、输入和急停日志。
- 不在缺少样本、路线定义或窗口前台证据时宣称准确率或完成。

## 4. 架构

```text
Windows Desktop / Target Window
  -> Capture Backend (DXGI/DXcam, fallback MSS)
  -> Frame Normalizer + Recorder + Replay
  -> Scene/HUD/Anchor Perception (OpenCV + ONNX/OCR)
  -> Localizer + Route Graph
  -> Behavior State Machine + Recovery
  -> Safety Gate
  -> DryRunActuator | Win32 SendInput Actuator
  -> PNG/JSONL Evidence + Offline Metrics
```

Python Worker 负责截图、视觉、回放、路线决策、Windows 输入和证据落盘；所有命令均从终端或 `.cmd` 一键入口运行。

## 5. 实施阶段

1. **Research**：GitHub/官方文档源码调研、Windows 硬件和软件基线、风险矩阵。
2. **Foundation**：Python 工程、配置模型、结构化日志、录制/回放、dry-run actuator。
3. **Capture**：DXcam/MSS backend、区域校准、性能基准和诊断图。
4. **Perception**：场景分类、HUD/锚点检测、OCR、数据标注与指标脚本。
5. **Navigation**：路线图、定位融合、行为状态机、偏航和卡住恢复。
6. **Input**：前台窗口守卫、标准 `SendInput`、急停、最大按键时长和人工接管。
7. **Integration**：纯 Python CLI、一键启动、运行状态和产物目录。
8. **E2E**：确定性测试窗口、离线回放、Windows 受控场景、独立评估。

## 6. 约束

- 不读取或扫描目标程序进程内存。
- 不注入 DLL、Hook 图形 API 或修改目标程序文件。
- 不实现内核驱动、硬件伪装、反作弊检测规避或封禁规避。
- 不把 HTTP 200、进程存活、模拟器测试或单张截图当作真实 E2E。
- armed 输入默认关闭，只有配置、窗口校验和急停自检通过后才能显式开启。
- 所有模型和第三方代码必须核对 License，权重来源必须可追溯。

## 7. 非目标

- 自动瞄准、自动射击、敌人识别或战斗决策。
- 账号登录、验证码处理、支付或交易自动化。
- 反作弊绕过、隐藏进程、随机化指纹或规避平台检测。
- 在没有真实样本和受控运行证据时承诺正式环境成功率。

## 8. 风险

- 游戏画面、HUD、分辨率和版本更新会造成模型或模板漂移。
- Windows 标准输入可能被目标程序忽略；本项目不会用更高风险的驱动/注入方式绕过。
- 小地图不可见、动态遮挡或转向速度不稳定会显著影响无内存定位精度。
- 在线使用可能违反目标产品条款并导致账号处罚；实现只在用户明确授权的受控环境中验证。

## 9. 进度记录规则

- `feature-list.json` 的新功能默认全部为 `passes: false`。
- 只有当前证据满足对应验收条件时才能改为 `true`。
- 每个阶段结束更新 `Codex-progress.txt`，记录命令、结果、产物、已知问题和下一步。
- 每个阶段独立提交，Windows 同步后再次验证。
